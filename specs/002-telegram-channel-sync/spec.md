**Updated**: 2026-05-02
**Status**: In Progress

## Clarifications

### Session 2026-04-30

### Session 2026-05-02

- Q: 业务流程确认 → A: 用户确认了两个核心业务流程：
  1. **首次配置流程**: 服务启动 → 添加配置 → 点击发送验证码 → 启动 Telegram 并发送验证码 → 登录成功 → 自动监听所有群组消息 → 配置频道 → 接收消息并创建文章
  2. **重启服务后自动恢复**: 服务重启 → 检查配置和认证状态 → 自动启动客户端 → 自动监听消息 → 接收配置频道的消息并创建文章

## User Scenarios & Testing *(mandatory)*

### User Story 0 - 首次配置流程 (Priority: P0) 🎯 核心流程

作为 CMS 管理员，我希望在服务启动后能够完成首次配置，包括填写 Telegram API 配置、发送验证码完成登录认证，认证成功后系统能够自动开始监听所有群组消息，然后我可以配置需要同步的频道，系统监听到对应频道的消息后能够自动创建文章。

**Why this priority**: 这是用户使用插件的第一步，没有这个流程用户无法使用任何功能。

**Independent Test**: 可以通过完整的配置流程验证：启动服务 → 配置 → 认证 → 监听 → 创建文章。

**Acceptance Scenarios**:

1. **Given** 服务已启动且插件已加载，**When** 管理员进入插件配置页面，**Then** 显示基本配置表单（App ID、App Hash、手机号等）
2. **Given** 管理员已填写手机号，**When** 点击"发送验证码"按钮，**Then** 系统启动 Telegram 客户端并发送验证码到用户手机
3. **Given** 用户已收到验证码，**When** 输入验证码并提交，**Then** 系统完成认证并保存会话，显示认证成功状态
4. **Given** 认证成功后，**When** 系统自动开始监听所有群组消息，**Then** 消息处理回调被正确设置
5. **Given** 管理员配置了要监听的频道，**When** 该频道发布新消息，**Then** 系统自动创建文章

---

### User Story 0.5 - 服务重启后自动恢复 (Priority: P0) 🎯 核心流程

作为 CMS 管理员，我希望在服务重启后，如果之前已经配置过并认证成功，系统能够自动启动 Telegram 客户端并恢复监听，无需我手动重新认证，对应频道接收到消息后能够自动创建文章。

**Why this priority**: 这是生产环境稳定运行的关键，没有自动恢复功能每次重启都需要重新认证，严重影响可用性。

**Independent Test**: 可以通过重启服务后验证：检查认证状态 → 检查监听状态 → 发送测试消息 → 验证文章创建。

**Acceptance Scenarios**:

1. **Given** 服务重启且存在已保存的会话数据，**When** 插件加载，**Then** 系统自动初始化 Telegram 客户端并尝试恢复会话
2. **Given** 会话恢复成功，**When** 客户端连接建立，**Then** 系统自动设置消息处理回调并开始监听
3. **Given** 会话恢复成功且已配置频道，**When** 配置的频道发布新消息，**Then** 系统自动创建文章
4. **Given** 会话已过期或失效，**When** 自动恢复失败，**Then** 系统显示需要重新认证的提示

---

### User Story 0.6 - 运行时动态添加频道 (Priority: P0) 🎯 核心流程

作为 CMS 管理员，我希望在系统正常运行时能够动态添加新的群组或频道到监听列表，添加后系统能够立即开始监听新添加的群组或频道，收到消息后能够自动创建文章，无需重启服务。

**Why this priority**: 这是实际使用中的常见场景，用户经常需要在运行时添加新的监听源。

**Independent Test**: 可以通过运行时添加新频道，发送测试消息，验证文章创建。

**Acceptance Scenarios**:

1. **Given** 系统正常运行且已认证，**When** 管理员添加新频道到监听列表，**Then** 新频道配置被保存
2. **Given** 新频道已添加，**When** 该频道发布新消息，**Then** 系统正确识别频道 ID 并创建文章
3. **Given** 系统正在监听多个频道，**When** 管理员添加新频道，**Then** 系统同时监听原有频道和新添加的频道
4. **Given** 管理员修改了频道配置，**When** 保存配置后，**Then** 新配置立即生效（通过重新解析 channels_json）

### User Story 1 - 频道消息自动同步发布 (Priority: P1)

作为 CMS 管理员，我希望能够配置一个或多个 Telegram 频道，当频道发布新消息时，系统能够自动将该消息转换为一篇文章并发布到 CMS 中，这样我可以减少手动复制粘贴的工作量。

**Why this priority**: 这是功能的核心价值，直接解决用户的主要需求——自动化内容同步。

**Independent Test**: 可以通过订阅一个测试频道，发送测试消息，验证文章是否在 CMS 中自动创建来独立测试。

**Acceptance Scenarios**:

1. **Given** 系统已配置监听频道 A，**When** 频道 A 发布新消息，**Then** 系统在 30 秒内创建一篇新文章，内容为该消息
2. **Given** 系统已配置多个监听频道，**When** 任一频道发布消息，**Then** 系统为每个频道创建独立文章
3. **Given** 频道消息包含图片，**When** 消息被同步，**Then** 图片被下载并作为文章封面或内容图片

---

### User Story 2 - 消息过滤规则配置 (Priority: P2)

作为 CMS 管理员，我希望能够设置过滤规则（如关键词白名单/黑名单、消息类型限制），只有符合规则的消息才会被发布，这样可以过滤掉不需要的内容（如广告、公告等）。

**Why this priority**: 过滤功能确保内容质量，但核心同步功能可以先独立工作。

**Independent Test**: 配置过滤规则后，发送符合和不符合规则的消息，验证只有符合规则的消息被发布。

**Acceptance Scenarios**:

1. **Given** 配置了关键词白名单 ["技术", "教程"]，**When** 频道消息包含任一关键词，**Then** 消息被发布为文章
2. **Given** 配置了关键词黑名单 ["广告", "推广"]，**When** 频道消息包含任一关键词，**Then** 消息被忽略
3. **Given** 配置了只同步包含图片的消息，**When** 频道发布纯文本消息，**Then** 消息被忽略
4. **Given** 配置了消息长度最小值 100 字符，**When** 消息长度不足，**Then** 消息被忽略

### User Story 3 - 频道配置管理 (Priority: P1)

作为 CMS 管理员，我希望通过管理后台界面添加、编辑、删除要监听的频道配置，包括频道 ID/链接、同步状态、目标分类等设置，这样我可以灵活管理同步源。

**Why this priority**: 配置管理是 P1 功能的必要支撑，没有它用户无法使用同步功能。

**Independent Test**: 通过管理界面完成频道的增删改查操作，验证配置是否正确保存和生效。

**Acceptance Scenarios**:

1. **Given** 管理员进入频道配置页面，**When** 添加新频道配置（频道 ID、名称、状态），**Then** 配置被保存并显示在列表中
2. **Given** 已有频道配置，**When** 管理员修改频道状态为"禁用"，**Then** 该频道停止同步消息
3. **Given** 已有频道配置，**When** 管理员删除频道配置，**Then** 配置被移除，历史同步的文章保留

### User Story 4 - 同步状态监控 (Priority: P3)

作为 CMS 管理员，我希望能够查看同步状态和日志，包括最近同步的消息、失败记录、连接状态等，这样我可以及时发现和解决问题。

**Why this priority**: 监控功能有助于运维，但不影响核心功能的正常使用。

**Independent Test**: 触发同步操作后，检查监控页面是否显示正确的状态信息。

**Acceptance Scenarios**:

1. **Given** 系统正在运行，**When** 管理员查看同步状态页面，**Then** 显示当前连接状态（已连接/断开/错误）
2. **Given** 同步过程中发生错误，**When** 管理员查看错误日志，**Then** 显示错误时间、频道、错误原因
3. **Given** 最近有消息同步成功，**When** 管理员查看同步记录，**Then** 显示最近 50 条同步记录及其状态

### User Story 5 - Telegram 认证与会话持久化 (Priority: P1)

作为 CMS 管理员，我希望首次完成 Telegram 认证后，即使插件或服务重启，系统也能自动恢复认证状态继续工作，无需我重新进行登录操作，这样我可以减少运维负担。

**Why this priority**: 认证持久化是插件稳定运行的基础，没有它每次重启都需要重新认证，严重影响可用性。

**Independent Test**: 完成首次认证后，重启服务，验证系统是否自动恢复连接并继续同步消息。

**Acceptance Scenarios**:

1. **Given** 管理员首次配置 Telegram 认证信息，**When** 认证成功，**Then** 会话信息被安全保存
2. **Given** 已有保存的会话信息，**When** 插件重启，**Then** 系统自动使用保存的会话恢复连接，无需重新认证
3. **Given** 会话信息已过期或失效，**When** 自动恢复失败，**Then** 系统提示管理员需要重新认证

---

### Edge Cases

- 当 Telegram 连接断开时，系统应该如何处理？(自动重连，记录错误日志)
- 当消息内容为空或仅包含无效媒体时，系统应该如何处理？(跳过该消息，记录日志)
- 当同一消息被重复接收时，系统应该如何处理？(通过消息 ID 去重，避免重复发布)
- 当频道被删除或设为私有后，系统应该如何处理？(检测错误并标记频道状态为异常)
- 当消息包含多种媒体类型（图片、视频、文件）时，如何处理？(优先处理图片，其他类型记录日志)
- 当消息超过文章内容长度限制时，如何处理？(自动截断或分拆为多篇文章)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须能够使用 gotd/td 库建立与 Telegram 服务器的持久连接
- **FR-002**: 系统必须能够订阅并监听一个或多个指定的 Telegram 频道
- **FR-003**: 系统必须在接收到新消息时，在 30 秒内将其转换为 CMS 文章
- **FR-004**: 系统必须支持配置消息过滤规则（关键词白名单/黑名单、消息类型、长度限制）
- **FR-005**: 系统必须能够处理消息中的图片媒体，下载并关联到文章
- **FR-006**: 系统必须提供管理后台界面用于配置监听频道
- **FR-007**: 系统必须支持启用/禁用单个频道的同步功能
- **FR-008**: 系统必须通过消息 ID 实现去重，避免重复发布
- **FR-009**: 系统必须记录同步日志，包括成功和失败的记录
- **FR-010**: 系统必须在连接断开时自动尝试重连
- **FR-011**: 系统必须支持为每个频道配置目标文章分类
- **FR-012**: 系统必须作为 Moss 插件实现，集成到现有 CMS 架构中
- **FR-013**: 系统必须支持 Telegram 认证状态持久化，首次认证成功后保存会话信息
- **FR-014**: 插件重启时，如果存在已保存的会话信息，必须能够自动恢复认证状态，无需用户重新登录 Telegram
- **FR-015**: 认证成功后，系统必须自动开始监听所有群组和频道的消息更新
- **FR-016**: 系统必须支持通过前端界面触发发送验证码流程，并启动 Telegram 客户端
- **FR-017**: 服务重启时，如果配置完整（App ID、App Hash、Session Key）且存在有效会话，必须自动启动 Telegram 客户端并恢复监听
- **FR-018**: 消息处理回调必须在认证成功后立即设置，确保新消息能够被正确处理
- **FR-019**: 系统必须能够区分频道消息和群组消息，并正确提取来源 ID
- **FR-020**: 系统必须支持运行时动态添加频道，新添加的频道配置保存后立即生效，无需重启服务
- **FR-021**: 消息处理时必须重新解析频道配置，确保使用最新的监听列表

### Key Entities

- **TelegramChannel（频道配置）**: 代表一个要监听的 Telegram 频道，包含频道 ID、名称、状态（启用/禁用）、目标分类、过滤规则、创建时间、最后同步时间
- **SyncRule（过滤规则）**: 代表消息过滤配置，包含规则类型（白名单/黑名单）、关键词列表、消息类型限制、最小长度
- **TelegramSession（会话信息）**: 代表 Telegram 认证会话，包含会话数据（加密存储）、创建时间、最后更新时间、状态（有效/过期）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新消息从发布到 CMS 文章创建的延迟不超过 30 秒
- **SC-002**: 系统支持同时监听至少 10 个频道而不影响性能
- **SC-003**: 消息同步成功率不低于 95%（排除被过滤的消息）
- **SC-004**: 管理员可以在 5 分钟内完成一个新频道的配置
- **SC-005**: 系统在连接断开后 60 秒内自动重连成功
- **SC-006**: 去重机制确保同一消息不会被重复发布

## Assumptions

- 用户已拥有 Telegram 账号并能够获取频道 ID 或链接
- 用户有权限访问要监听的频道（公开频道或已加入的私有频道）
- CMS 后端服务运行稳定，能够接收文章创建请求
- 网络环境能够正常访问 Telegram 服务器
- 使用现有的 CMS 文章数据模型，不创建新的内容类型
- 消息中的视频和文件类型暂不处理，仅记录日志
- 同步的文章默认状态为"已发布"，可通过配置修改
- Telegram 会话信息使用加密存储，确保安全性
- 插件遵循 Moss 现有的插件架构和生命周期管理

## Implementation Status

### 已实现功能 ✅

1. **认证流程 (FR-016)**
   - `SendAuthCode`: 发送验证码并启动客户端
   - `VerifyAuthCode`: 验证验证码并完成认证
   - 前端界面支持发送验证码和验证流程

2. **会话持久化 (FR-013)**
   - `DBStorage` 实现了 `session.Storage` 接口
   - 使用 AES-256-GCM 加密存储会话数据
   - 会话数据存储在 `telegram_session` 表

3. **服务重启自动恢复 (FR-014, FR-017)**
   - `initClient` 中检查 session storage 并自动启动客户端
   - 会话恢复成功后自动设置 `Authenticated` 和 `Connected` 状态
   - 消息处理回调在客户端初始化时设置

4. **消息监听 (FR-015, FR-018)**
   - `handleUpdate` 处理所有 Telegram 更新
   - `handleNewChannelMessage` 处理频道消息
   - `handleNewMessage` 处理群组消息
   - 消息处理回调通过 `SetMessageHandler` 设置

5. **频道配置管理 (FR-006, FR-007)**
   - 前端支持添加、编辑、删除频道配置
   - 频道配置通过 `channels_json` 字段存储
   - 支持从 Telegram 获取频道列表

6. **文章创建 (FR-003)**
   - `handleChannelMessage` 中调用 `CreateArticle` 创建文章
   - 支持消息去重 (FR-008)
   - 记录同步日志 (FR-009)

### 业务流程验证

#### 场景 1: 首次配置流程 ✅
```
服务启动 → 插件加载 (Load) → 
管理员配置 App ID/App Hash/手机号 → 
点击发送验证码 (SendAuthCode) → 
客户端启动并发送验证码 → 
输入验证码 (VerifyAuthCode) → 
认证成功，会话保存 → 
配置频道 → 
消息监听已就绪 → 
收到消息 → 创建文章
```

#### 场景 2: 服务重启后自动恢复 ✅
```
服务重启 → 插件加载 (Load) → 
initClient 检查 session storage → 
自动启动客户端 (Start) → 
会话恢复成功 → 
消息处理回调已设置 → 
收到配置频道的消息 → 创建文章
```

#### 场景 3: 运行时动态添加频道 ✅
```
系统正常运行 → 管理员添加新频道 → 
前端调用 pluginSaveOptions → 
后端 UpdateOptions 保存配置并更新插件内存 → 
p.ChannelsJSON 被更新 → 
收到新频道消息 → 
handleChannelMessage 调用 parseChannels() 解析最新配置 → 
找到频道配置 → 创建文章
```
