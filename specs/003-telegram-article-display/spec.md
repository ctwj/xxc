## 前端展示策略

### 内容类型映射

根据 `InfoCardData` 类型定义：

| 类型 | 条件 | 展示布局 |
|------|------|----------|
| `text` | 纯文本，无媒体 | 居中文本卡片 |
| `long-text` | 文本 > 500字符 | 长文阅读器 |
| `image` | 单图无文 | 全屏图片+渐变标题 |
| `image-text` | 单图+文 | 60%图片+40%文字 |
| `images` | 多图无文 | 图片轮播 |
| `images-text` | 多图+文 | 图片轮播+文字区 |
| `video` | 视频无文 | 全屏视频播放 |
| `video-text` | 视频+文 | 65%视频+35%文字 |

### 展示字段策略

#### 必须展示
1. **标题** (`title`): 卡片顶部或叠加显示
2. **发布时间** (`create_time`): 相对时间格式（如"2小时前"）
3. **内容** (`content`/`description`): 根据类型截断或完整展示

#### 可选展示
4. **缩略图** (`thumbnail`): 作为主图或封面
5. **媒体集合** (`media_urls`): 图片轮播
6. **视频** (`video_url`): 内嵌播放器
7. **分类标签** (`category_id`): 显示分类名称

#### 暂不展示
- `slug`: 仅用于 URL 路由
- `views`: 浏览量（可考虑在详情页展示）

### 布局决策

#### 当前数据适配
由于同步的文章当前无媒体文件，前端应：

1. **自动检测类型**:
   ```typescript
   if (!article.thumbnail && !article.mediaUrls && !article.videoUrl) {
     type = article.content?.length > 500 ? 'long-text' : 'text';
   }
   ```

2. **文本卡片优化**:
   - 短文本（< 150字）: 居中大字展示
   - 长文本（> 150字）: 使用 `LongTextReader` 组件

3. **标题处理**:
   - 无标题时显示"来自 Telegram 的消息"
   - 标题过长时截断并添加省略号

### 媒体增强需求

#### 后端改进
1. **关联媒体到文章**:
   - 将下载的图片 URL 写入 `thumbnail` 字段
   - 多图时写入 `media_urls` JSON 数组
   - 视频写入 `video_url` 字段

2. **内容格式优化**:
   - 图片消息: 内容保持原文，媒体单独存储
   - 图文消息: 内容保留文字部分

#### 前端适配
1. **媒体优先级**:
   - 有视频时优先展示视频
   - 多图时展示轮播
   - 单图时展示图文卡片

2. **降级策略**:
   - 媒体加载失败时显示纯文本
   - 视频播放失败时显示封面图

## 功能需求

### FR-1: 媒体关联增强
**描述**: 同步消息时正确关联媒体文件到文章字段

**验收标准**:
- 图片消息的 `thumbnail` 字段包含第一张图片 URL
- 多图消息的 `media_urls` 字段包含所有图片 URL JSON 数组
- 视频消息的 `video_url` 字段包含视频 URL
- `telegram_media` 表正确记录媒体元数据

### FR-2: 标题生成优化
**描述**: 改进标题生成逻辑，支持更多场景

**验收标准**:
- 纯图片消息: 使用"来自 {频道名} 的图片"或时间戳
- 空消息: 使用"来自 {频道名} 的消息"
- 长文本: 保持首行截取，确保语义完整

### FR-3: 前端类型自动检测
**描述**: 前端根据文章字段自动判断展示类型

**验收标准**:
- 无媒体时正确显示文本卡片
- 有媒体时正确显示对应媒体类型
- 类型判断逻辑与后端数据结构一致

### FR-4: 长文本阅读体验
**描述**: 优化长文本内容的展示和阅读体验

**验收标准**:
- 超过 500 字符自动使用 `long-text` 类型
- 支持滚动阅读
- 保持段落格式
